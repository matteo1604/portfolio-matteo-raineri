# CLAUDE.md

## Project identity
This is not a standard portfolio.

It is an interactive system composed of:
- a capability engine (Skills section)
- an execution system (Process section)
- a narrative manifesto (Philosophy)
- a modular archive system (BuildArchive)
- layered UI primitives (FloatingElements, Cursor, Transitions)

The goal is not to display projects.
The goal is to let the user experience how the system thinks.

## Current phase
The project is in FINALIZATION phase.

This means:
- no redesign from scratch
- no architectural reset
- no visual identity changes

Only:
- refinement
- cohesion
- animation orchestration
- system-level polish

## Core principle
Elevate, do not reinvent.

Every change must:
- increase perceived quality
- improve cohesion
- enhance motion language
- preserve identity

## Key insight (CRITICAL)
This project is already strong.

The problem is NOT:
- missing features
- missing complexity

The problem is:
- disconnected sections
- underutilized motion system
- lack of orchestration between parts

## Priority order (strict)
When improving the project, ALWAYS follow this order:

1. Section cohesion
2. Motion orchestration
3. Section transitions
4. Micro-interactions
5. Visual refinement
6. Code cleanup

NOT the opposite.

## Editing rules
Before changing anything:

1. analyze the component
2. identify what already works
3. identify the weakest part
4. improve only that part

Never:
- rewrite entire sections blindly
- change design direction randomly
- add complexity without reason

## Animation philosophy
Animations must feel:

- controlled
- intentional
- structural
- premium

NOT:
- flashy
- random
- excessive
- decorative only

## Performance rules
Always protect:

- smoothness
- responsiveness
- readability

Avoid:
- heavy blur abuse
- excessive DOM updates
- layout thrashing

## Code rules
- keep TypeScript strict
- keep components readable
- avoid giant files unless justified
- reuse patterns when useful
- avoid duplication

## When proposing improvements
Always respond with:

1. audit of current component
2. main issue
3. improvement strategy
4. implementation
5. why it improves the system

## Definition of "better"
A change is better ONLY if it improves:

- cohesion between sections
- motion consistency
- system readability
- perceived quality
- performance
- interaction clarity

If not, do not change it.
